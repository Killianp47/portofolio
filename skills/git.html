Git

GIT est un système de contrôle de version distribué largement utilisé dans le développement logiciel, créé par Linus Torvalds en 2005 pour la gestion du noyau Linux. Au fil des années, il est devenu un pilier fondamental dans le processus de développement collaboratif, offrant un ensemble robuste d'outils pour suivre les modifications de code source, coordonner le travail d'équipes dispersées et gérer les différentes branches de développement.

Les avantages de GIT sont nombreux. Sa capacité à permettre le travail collaboratif simultané sur un même projet, tout en conservant un historique complet des modifications apportées, en fait un outil précieux pour les équipes de développement distribuées. De plus, sa structure décentralisée permet aux développeurs de travailler en mode déconnecté, ce qui est particulièrement utile dans les environnements où la connectivité réseau peut être limitée.

Bien que GIT offre de nombreux avantages, il existe également des défis à relever. La courbe d'apprentissage initiale peut être raide pour les nouveaux utilisateurs, en particulier ceux qui n'ont pas d'expérience préalable avec les systèmes de contrôle de version. Comprendre les concepts fondamentaux tels que les commits, les branches et les fusions peuvent demander du temps et de la pratique. De plus, la gestion efficace des conflits lors de la fusion de branches peut être complexe, notamment dans les projets de grande envergure avec de multiples contributeurs. Une compréhension approfondie des stratégies de fusion et des outils de résolution de conflits est essentielle pour maintenir l'intégrité du code source et éviter les perturbations dans le processus de développement.

Dans mon propre parcours professionnel, j'ai eu l'occasion d'explorer GIT dans le cadre de divers projets de développement logiciel. Grâce au projet BL.PMSI, j'ai pu apprécier la valeur de GIT en tant qu'outil de collaboration et de gestion de code source. Notamment, il est essentiel de prêter une attention particulière à la configuration de GIT sur l'environnement local, comme j'ai pu le constater sur le projet BL.PMSI qui utilise plusieurs workflows distincts, notamment les branches develop, release et master. La branche develop constitue l'environnement où tous les développeurs effectuent leurs modifications. La branche release est dédiée à la création d'une image Docker et à la version de l'application pour les tests. Enfin, la branche master permet, une fois le développement testé, de libérer la version pour qu'elle soit disponible pour les clients. Cependant, une particularité importante réside dans le processus de transition de la branche release à la branche master. Pour ce faire, il est nécessaire de modifier les paramètres du référentiel, plus précisément des demandes de fusion (merge requests), afin de les configurer en tant que « merge commit ». Malheureusement, dans mon environnement local mal configuré, les merge commits étaient également présents lors de l'utilisation de la commande de rebase. Cela a créé des incohérences entre les différents développements qui étaient sur la branche release.


Cette expérience m'a permis de comprendre l'importance cruciale de la configuration appropriée de GIT, en particulier dans des environnements complexes avec des workflows spécifiques. Cela souligne également la nécessité d'une compréhension approfondie des fonctionnalités avancées de GIT et des bonnes pratiques de gestion des branches pour assurer la cohérence et la fiabilité du processus de développement logiciel.

Malgré mon niveau de compétence actuel sur GIT dans lequel je suis autonome, je suis conscient qu'il y a toujours place à l'amélioration. En explorant des fonctionnalités avancées telles que les hooks, les rebases interactifs et les submodules, je peux approfondir ma compréhension de GIT et l'intégrer plus efficacement dans mes flux de travail de développement.
